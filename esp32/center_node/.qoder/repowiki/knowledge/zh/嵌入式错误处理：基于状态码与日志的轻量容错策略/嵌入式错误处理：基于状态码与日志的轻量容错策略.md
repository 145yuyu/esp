本仓库为 ESP32-S3 中心节点固件，采用 Arduino/ESP-IDF 混合开发模式。代码库中**未定义统一的错误类型、异常机制或 panic/recover 体系**，错误处理以“返回码 + Serial 日志 + 局部恢复”为主，属于典型的嵌入式轻量级风格。

### 1. 使用的系统/方法
- **HTTP 层**：使用 `HTTPClient::errorToString(code)` 将底层错误码转为可读字符串，通过 `Serial.printf` 输出；无结构化错误对象。
- **ESP-NOW 协议层**：通过长度校验 + CRC16 校验（`calc_crc16`）检测数据完整性，失败时直接丢弃并打印 `[RX]` 日志。
- **传感器读取**：DHT22 返回 NaN 时回退到上一周期值（`g_prev_temp/g_prev_hum`），不向上抛错。
- **初始化阶段**：互斥锁创建失败进入 `while(1) delay(1000)` 死循环；ESP-NOW 初始化失败仅打印日志并在屏幕显示提示，不影响主流程。
- **WiFi 连接**：STA 连接超时后继续运行 AP 模式，后台周期性重试，不中断系统启动。

### 2. 关键文件与位置
- `main/main.cpp`：全部错误处理逻辑集中于此，包括 HTTP 上传失败、ESP-NOW 接收校验失败、传感器容错、初始化失败等。
- `src/main.cpp`：与 `main/main.cpp` 结构相同的另一份入口，包含相同错误处理模式。
- `protocol.h`（被 include 但未在仓库中可见）：定义了 `sensor_data_t`、`AlertLevel`、`ANOMALY_*` 标志位及 `gp2y_calc_dust` 等函数，是上层业务错误语义的来源。

### 3. 架构与约定
- **无统一错误码枚举**：各模块自行判断错误（如 `WiFi.status() != WL_CONNECTED`、`isnan(t)`、CRC 不等），没有跨模块共享的错误码表。
- **告警等级作为“软错误”**：`AlertLevel`（NORMAL/WARNING/DANGER/CRITICAL）和 `anomaly_flags`（`ANOMALY_SENSOR_ERR`、`ANOMALY_TEMP_HIGH` 等）用于表达可恢复的业务异常，驱动 LED/蜂鸣器，而非中断控制流。
- **FreeRTOS 任务隔离**：传感器、通信、显示、告警各自独立任务，通过互斥锁保护共享数据；任务内部错误不会导致其他任务崩溃。
- **WebServer 中间件缺失**：HTTP 路由回调内直接 `server.send(...)`，无全局错误处理器或中间件统一捕获异常。

### 4. 开发者应遵循的规则
- **不要抛出 C++ 异常**：嵌入式环境禁用 RTTI/异常，所有错误通过返回值、布尔标志或状态字段传递。
- **对外部依赖（WiFi/ESP-NOW/传感器）调用必须判错**：参考现有模式——检查返回值/状态后立即记录 `Serial.printf` 日志并降级处理。
- **业务异常用 `anomaly_flags` 和 `AlertLevel` 表达**：新增异常场景应在 `protocol.h` 中扩展标志位，并在 `check_all_alerts()` 中映射到告警等级。
- **初始化失败需保证系统仍可运行**：参考互斥锁/ESP-NOW 初始化失败的处理方式，避免 `abort()` 或 `panic()`，至少提供降级路径。
- **HTTP 错误统一走 `errorToString`**：新增网络请求时应复用此模式，避免硬编码错误消息。
